## 2026-03-19 — stability: Error handling and retry logic

**Completed:**
- Created `apps/api/src/lib/agent-errors.ts` with typed error classes (`AgentLoopError`), error codes (LLM_TIMEOUT, LLM_RATE_LIMIT, LLM_MALFORMED_RESPONSE, API_NETWORK_ERROR, etc.), and `classifyLlmError` / `classifyApiError` / `logStructuredError` utilities.
- Updated `apps/api/src/services/llm-router.ts`: added per-model retry with exponential backoff (up to 2 retries for primary models, 1 for emergency). Transient errors (timeout, rate-limit, network) trigger retry with 500ms/1000ms delays. Non-transient errors skip to next model immediately.
- Updated `apps/api/src/agents/agent-loop.ts`: GeckoTerminal and DexScreener API failures now emit structured error log lines (`error_code=… recoverable=… message=…`). LLM call failures also use structured logging and always persist a hold decision to D1.
- Added `apps/api/tests/stability.test.ts` with 24 tests covering error classification, context propagation, log format, and recovery flags.

**Notable decisions:**
- Used `maxRetries: 0` in `generateText` to keep retry logic fully in our control.
- Per-model retry + model fallback chain = two-level resilience.
- Pre-existing failures (phase1 test + web lint) are unrelated to this task.

**Next recommended:** observability or testing (expand e2e agent lifecycle).

## 2026-03-19 — observability: Logging, metrics, and debug endpoint

**Completed:**
- Created `apps/api/src/lib/logger.ts` with `createLogger(service, agentId?)` that emits structured `key=value` log lines with level, timestamp, service, and optional agentId. Includes `log.time(event)` for automatic duration logging — returns elapsed ms.
- Updated `apps/api/src/agents/agent-loop.ts`: added `tick_start`, `analysis_start`, `market_data_fetch` (timed), `price_resolution_failed` (warn), `indicator_compute` (timed per-pair), `llm_call` (timed), `decision` (with confidence/tokens/latency), `trade_open`, `trade_close`, and `tick_end` (total duration) structured log events.
- Updated `apps/api/src/agents/trading-agent.ts`: added `/debug` endpoint that returns full DO state snapshot (agentId, status, analysisInterval, nextAlarmAt, tickCount, isLoopRunning, lastStopOutAt, pendingTrade, priceMiss counters, engine summary).
- Updated `apps/api/src/routes/agents.ts`: added `GET /api/agents/:id/debug` route gated to tester role (403 for non-testers) — proxies to DO `/debug` endpoint.
- Added `apps/api/tests/observability.test.ts` with 13 tests covering log format, field encoding, level routing, timestamp format, null field filtering, and timer accuracy.

**Notable decisions:**
- Log format is key=value (not JSON) for easy `wrangler tail` grep — strings are JSON-quoted, numbers are plain.
- Timer returns elapsed ms so callers can use the value (e.g. llmDurationMs for the decision log).
- Debug endpoint exposes DO internals only to tester-role users — prevents leaking engine state to regular users.
- Pre-existing failures (phase1 test + web lint) are unrelated to this task.

**Next recommended:** architecture (DO as source of truth) or testing (expand e2e agent lifecycle).

## 2026-03-19 — orchestration: AgentManagerDO centralized scheduling

**Completed:**
- Extended `apps/api/src/agents/agent-manager.ts`: added scheduler endpoints (`/scheduler/register`, `/scheduler/unregister`, `/scheduler/agents?interval=X`, `/scheduler/list`, `/scheduler/sync`) to `AgentManagerDO`. Uses `idFromName('scheduler')` for a single global instance. Registry stored as JSON object (`schedulerAgents` key) in DO storage.
- Updated `apps/api/src/routes/agents.ts`: added `notifyScheduler()` helper (best-effort, non-fatal). Calls `register` on start/patch-interval, `unregister` on stop/pause/reset/delete.
- Refactored `apps/api/src/index.ts` cron handler: added `getScheduledAgentIds()` that calls scheduler DO first. Falls back to full D1 scan if DO is unavailable. Snapshot-only cron (`0 * * * *`) no longer returns early — now also triggers 1h agents.
- Added `apps/api/tests/orchestration.test.ts` with 14 tests covering registry CRUD, interval matching, legacy interval normalization, sync rebuild, and endpoint input validation.

**Notable decisions:**
- Used `AGENT_MANAGER.idFromName('scheduler')` — reuses existing binding without new wrangler.toml changes.
- Scheduler notifications are best-effort (never block the HTTP response) — D1 fallback ensures correctness if scheduler state is stale.
- Registry is a simple `Record<agentId, interval>` JSON object — cheap to read/write, trivially iterable by interval.
- Pre-existing phase1 test failure and web lint errors are unrelated to this task.

**Next recommended:** architecture (DO as source of truth) or security (API hardening).
